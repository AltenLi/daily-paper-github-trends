# 3类 OpenClaw 任务别再交给大模型：省 token 的第一张清单

这篇文章面向搜索 OpenClaw、AI Agent、个人自动化助手、权限安全和自动化工作流的读者。

很多人第一次用 Agent，会把重点放在 prompt 上。但真正能长期跑起来的系统，靠的不是一句神奇提示词，而是工作流、权限、日志、验收和复盘。

## 1. 问题场景

一个人做内容、研究、数据和自动化时，最容易出现的问题不是“模型不会写”，而是任务状态混乱：资料是谁收的，事实是谁核的，草稿是谁改的，发布前谁负责检查，失败后谁记录原因。

如果这些都靠一个聊天窗口记忆，短期看很灵活，长期一定会乱。上下文会丢，职责会重叠，失败也很难复盘。

## 2. 工作流拆解

我更推荐把 Agent 当成一个小团队来设计：

- 资料准备：只负责收集和入库，不负责下结论。
- 选题排序：只负责判断优先级，不负责编事实。
- 写作生产：只负责把材料组织成文章。
- 事实核查：区分事实、推断、疑点。
- 反方审稿：专门找漏洞、空话和风险。
- 格式检查：处理公众号 HTML、封面、标题、列表和链接。
- 发布准备：只准备草稿，不自动发布。

这套分工的价值是：每一步都能留下文件和状态，而不是靠“我感觉已经做完了”。

## 3. 为什么不能只靠 prompt

Prompt 可以告诉模型“认真一点”，但它不能替代状态机。

比如日更流程里，如果某篇文章还只是筛选稿，正确动作应该是返回写作和核查，而不是为了凑数量强行进入草稿箱。这个判断必须由 gate 和状态字段完成。

再比如发布动作。写稿、生成草稿、正式发布是三个权限等级。Agent 可以自动写稿，可以准备草稿，但正式发布必须人工确认。

## 4. 失败案例

典型失败有三种。

第一，父目录污染路由。目录名里有 GitHub，导致论文也被当成 GitHub 稿。解决方法是路由只看 basename 和正文证据，不看父目录。

第二，公众号 HTML 列表渲染异常。Markdown 列表转成真实 li 标签后，微信草稿可能出现空列表。解决方法是公众号正文使用段落型伪列表，并增加结构 gate。

第三，生产 owner 混乱。CEO/总控如果亲自做每条生产流，系统会变成单点瓶颈。解决方法是把生产状态机交给生产制片人，CEO 只做验收和升级确认。

## 5. 安全边界

公开文章不能暴露账号、浏览器凭证、内网地址、真实聊天、私有路径和不可逆操作脚本。

自动化系统也不能越过人工确认去做发布、删除、真实交易、转账或外部联系。

这不是保守，而是让 Agent 能长期可靠运行的底线。

## 6. 可复用 checklist

- 每个任务是否有 owner。
- 每一步是否有输入和输出文件。
- 是否能从日志知道失败在哪。
- 是否区分草稿、外发和正式发布。
- 是否有事实核查和反方审稿。
- 是否有格式 gate。
- 是否把敏感信息排除在公开稿之外。

## 7. 读者可以怎么照搬

最小版本不需要搭复杂公司。只要给自己的 Agent 建四个文件夹：materials、drafts、reviews、publish-ready，再配一个状态字段：planned、drafted、checked、blocked、ready。

当任务 blocked 时，不要手动忽略它，而是写清楚 blocker，然后让下一轮自动返修。这样系统会越来越稳。
