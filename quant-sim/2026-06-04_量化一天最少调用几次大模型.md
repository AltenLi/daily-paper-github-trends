# 量化一天最少调用几次大模型

这篇文章属于「AI 量化模拟仓 / 信息工程」连续专栏。它不是荐股，不是带单，也不承诺收益；它只讨论怎样让 OpenClaw 这类 Agent 在高噪声市场里做 **记录、证据分层、风控提醒和复盘**。

今天换一个更本质的问题：如果我想把 AI 接进量化/模拟交易流程，**一天到底最少需要调用几次大模型？**

我的答案是：

> 对多数个人量化和模拟盘系统来说，常规交易日最少 0 次，标准配置 3 次，异常行情才临时加 call。

真正应该高频运行的是脚本、规则、行情刷新、风控闸门和日志系统；大模型只应该在“需要解释、归因、权衡和改规则”的时刻出现。

## 1. 先把问题问对

很多人一做 AI 量化，就会自然想到：

- 每 5 分钟让大模型看一次行情？
- 每条新闻都让大模型判断利好利空？
- 每次持仓波动都问一句“要不要卖”？

这听起来很智能，实际很危险。

第一，token 成本会失控。市场没给你信号，模型却一直在给你语言输出。

第二，判断会被噪声放大。5 分钟级别的波动，大多数只是市场呼吸，不值得升级成“策略观点”。

第三，系统会变成聊天机器人，而不是量化系统。量化系统最重要的是可复盘、可回测、可验证；如果每一步都靠大模型临场发挥，最后很难知道到底是哪条规则有效。

所以更好的问题不是“多久问一次大模型”，而是：

> 哪些事情必须交给大模型，哪些事情脚本就够了？

## 2. 一天 0 次：什么时候完全不用大模型

先说最小答案：**有些交易日，大模型一次都不用调用。**

如果满足这些条件，脚本就够了：

- 没有新持仓，或者持仓只是常规波动。
- 没有重大新闻、公告、龙虎榜、宏观冲击。
- 策略规则没有触发买入、卖出、止损、降仓、换仓。
- 回撤、仓位、成交量、波动率都在预设阈值内。
- 当天只是执行既定规则，不需要解释新情况。

这种情况下，大模型不应该为了“看起来在工作”而输出观点。

更合理的做法是让脚本每 5 分钟刷新：

- 行情快照
- 持仓市值
- 现金和仓位
- 回撤
- 规则触发状态
- 新闻标签计数
- 异常事件标记

如果没有触发异常，系统只写日志，不调用模型。

这就是低 token 架构的第一原则：

> 能被规则稳定描述的事情，不要交给大模型。

## 3. 一天 1 次：只做收盘复盘

如果系统还在早期，最小可用版本可以只在收盘后调用一次大模型。

这一次不负责盘中决策，只负责复盘：

- 今天策略有没有按规则运行？
- 哪些信号触发了，哪些没有触发？
- 新闻和行情是否互相验证？
- 有没有错误归因？
- 有没有纪律问题？
- 明天需要修改哪一条规则？

这个配置适合三类人：

- 刚开始搭模拟盘，不想让 AI 干预盘中判断。
- 主要做中低频策略，不需要盘中解释。
- 更关心系统迭代，而不是日内操作。

一天 1 次的好处是成本最低，也最不容易让模型制造交易冲动。

它的缺点也明显：盘中突发事件只能靠规则和人工处理，模型无法参与实时解释。

## 4. 一天 3 次：个人量化的标准配置

如果你希望 Agent 真正参与“观察市场”，我认为标准配置是一天 3 次。

不是因为 09:30、10:30、14:30 这三个时间有神秘魔法，而是因为它们分别对应三种不同的问题。

### 第一次：开盘后，只看风险，不做预测

开盘附近信息最乱。隔夜新闻、集合竞价、昨日情绪、开盘资金都会挤在一起。

这时候大模型最不该做的是预测全天走势。

它应该只回答：

- 隔夜有没有新增重大信息？
- 持仓有没有直接相关风险？
- 新闻是否已被市场初步反映？
- 是否有必须立刻人工确认的异常？
- 今天哪些规则需要打开观察？

输出最好是结构化的：

```json
{
  "checkpoint": "open_risk_scan",
  "need_llm": true,
  "market_context": "隔夜新闻和开盘表现是否冲突",
  "risk_flags": [],
  "rules_to_watch": [],
  "human_required": false
}
```

这一 call 的目标不是赚钱，而是避免带着错误前提开盘。

### 第二次：第一小时后，看主题是否稳定

第一小时之后，市场通常已经给出一些答案。

这时候大模型可以做的是“解释结构”，不是“追涨杀跌”。

它要看：

- 早盘强势主题是否延续？
- 资金是在扩散，还是只集中在少数标的？
- 新闻标签和盘面表现是否一致？
- 持仓所在主线是否仍然成立？
- 有没有从“个股波动”升级为“主线退潮”的证据？

这里要特别注意：如果持仓仍在主线里，且没有明显破位、没有大资金流出、没有主线退潮，就不要因为日内小波动频繁加减仓。

大模型的价值是帮你区分：

- 这是噪声；
- 这是结构变化；
- 这是风险升级。

### 第三次：尾盘前，做不确定性管理

尾盘 call 不应该问“明天涨不涨”。

它应该问：

- 今天的收益/回撤是否来自规则内行为？
- 有没有隔夜风险需要降级处理？
- 明天需要观察哪些证据？
- 今天哪些判断要进入复盘 ledger？
- 哪些规则需要保留、暂停或改写？

尾盘是最适合大模型的时间点，因为它不用在最嘈杂的时候做即时反应，而是基于一天的证据做归纳。

换句话说：

> 盘中脚本负责守纪律，尾盘模型负责改进纪律。

## 5. 异常加 call：什么时候可以临时调用

一天 0/1/3 次是常规配置。真正需要临时加 call 的，是异常事件。

比如：

- 持仓相关公司突发公告。
- 宏观新闻明显冲击利率、汇率、商品或风险偏好。
- 板块突然从扩散变成退潮。
- 策略触发了原本没有覆盖的新场景。
- 多个数据源互相矛盾，需要做事实分层。
- 系统检测到异常回撤、异常成交量或异常新闻密度。

但临时调用也要有门槛，不能“有点波动就问”。

我更推荐写成规则：

```json
{
  "extra_llm_call_required": true,
  "trigger": "news_conflict_or_risk_threshold",
  "evidence": [
    "新闻源A与新闻源B结论冲突",
    "持仓相关标签在30分钟内密集出现",
    "回撤超过预设阈值"
  ],
  "expected_output": "事实分层 + 风险等级 + 是否需要人工确认"
}
```

也就是说，大模型不是随叫随到的情绪安慰，而是异常处理器。

## 6. 今天的材料怎么放进这个框架

以今天的模拟盘材料为例，脚本已经能稳定做这些事：

- 读取权益、现金、持仓市值和浮动盈亏。
- 汇总新闻标签：半导体/光刻机、中东/能源/通胀、AI 光通信/DCI、电力电网、期货商品等。
- 生成收益曲线。
- 记录 09:30、10:30、14:30 的策略检查文件。

这些不需要大模型高频参与。

大模型真正应该看的，是脚本整理后的摘要：

- 今天新闻和盘面是否互相印证？
- 持仓所在主线是否还成立？
- 风险字段有没有缺失？
- 复盘字段有没有缺失？
- 下一轮要改哪一条规则？

今天暴露出一个典型问题：检查点里已经有新闻上下文和风险/失效条件，但复盘字段还不够完整。

这说明下一步不是增加更多 call，而是先把复盘 schema 补齐。

## 7. 一个可照搬的最低调用策略

如果你要搭自己的 AI 量化 Agent，可以直接从这个版本开始。

### 默认不调用大模型

脚本每 5 分钟运行一次：

- 行情快照
- 持仓对照
- 风控阈值
- 新闻标签
- 规则触发
- 日志落盘

没有异常，就不调用。

### 标准日调用 3 次

- 开盘：风险扫描，不预测。
- 第一小时：主题稳定性检查，不追涨杀跌。
- 尾盘：不确定性管理和复盘准备。

### 异常日追加调用

只有当规则触发异常，才临时调用：

- 新闻冲突
- 持仓重大公告
- 主线资金退潮
- 价格明显破位
- 回撤超过阈值
- 数据源矛盾

### 收盘后必须沉淀

无论当天调用几次，最后都要写入：

- decision-ledger
- risk-ledger
- rule-change-log
- error-sample
- next-watch-list

否则大模型说得再多，也不会让系统变聪明。

## 8. 判断一次调用是否值得

每次调用大模型前，可以问 5 个问题：

1. 这件事脚本能不能判断？
2. 是否有新信息需要解释？
3. 是否涉及多个来源冲突？
4. 是否会影响规则、仓位或风险等级？
5. 这次输出能不能写进复盘文件？

如果 5 个问题里只有第 1 个成立，那就不要调用。

如果第 2、3、4、5 都成立，那这次调用大概率值得。

我会把它写成一句简单原则：

> 大模型不负责盯盘，大模型负责解释“为什么规则要变”。

## 9. 风险边界

这套方法不能预测市场，也不能把 AI 变成交易责任人。

它能做的是：

- 减少无意义 token 消耗。
- 降低盘中噪声交易。
- 让新闻、行情、持仓、规则之间可追踪。
- 把错判沉淀成可复盘样本。
- 让系统从“会聊天”变成“会留证据”。

真实交易必须由人独立判断。Agent 可以做记录、提醒、核查和复盘，但不能替代责任。

## 10. 下一步

下一篇更适合接着讲：**如何设计一次 LLM 调用的输入包**。

不是把所有行情、新闻、持仓原文都塞给模型，而是把它们压缩成：

- 当前状态
- 触发规则
- 证据摘要
- 冲突信息
- 风险阈值
- 需要模型回答的问题
- 必须输出的 JSON schema

这才是低 token 量化系统真正能长期跑下去的关键。
